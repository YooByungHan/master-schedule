## 1. 버전별 데이터 저장 구조

### 1-1. Pro 버전 (로컬 단독)

| 항목 | 내용 |
|------|------|
| 저장 위치 | 브라우저 localStorage |
| 사용자 구분 | 없음 — 같은 PC·브라우저면 동일 저장소 공유 |
| 회사명 | 로그인 후 **업체명 칸에서 직접 입력·수정** (localStorage에 기억) |
| 보안 수준 | 엑셀 파일과 동일 — 해당 PC 접근자는 누구나 읽기·수정 가능 |
| 저장 파일 | `MasterSchedule백업_<회사명>_YYYYMMDD.json` |

**파일 portability(중요)**: Pro 저장 파일은 **현재 회사 1개만** 담고, 식별 정보(`primaryId`, `companyName`)를 함께 기록합니다. 따라서 **다른 사람이 열어도 그 파일의 공정표가 그대로** 열립니다(만든 사람 이름과 무관). 여는 쪽은 `primaryId → 파일명/회사명 일치 → 노드 최다` 순으로 진짜 공정표를 선택합니다.

### 1-2. 서버(Enterprise) 모드 — 멀티 현장 구조

서버 모드는 단일 `data.json`이 아니라 **현장(공사)별로 데이터가 분리**됩니다.

| 파일 | 내용 |
|------|------|
| `accounts.json` | 전사 계정·역할·현장 배정 — `{users:[{id,name,pw,role,company,sites[],adminCode}], sites:[{id,name,managerIds[],primaryCompanyId}], defaultManagerCode}` |
| `data/<siteId>.json` | **현장별 공정표** — `{companies:[...], projects:{...}}` (원도급 + 취합된 하도 슬롯) |
| `config.json` | Claude/Groq API 키 — 서버에만 존재, 브라우저로 절대 미전송 |

```
data/site_001.json = {
  companies: [
    { id: '원도급ID', name: '동부건설(주)' },   // 주 공정표(primaryCompanyId)
    { id: '하도ID', name: '한빛전기(주)' }, ...   // 취합/회의로 들어온 하도 슬롯
  ],
  projects: {
    '원도급ID': { sections:[...], ... },
    '하도ID':   { sections:[...], ... }
  }
}
```

- **실시간 동기화**: WebSocket — 한 명이 저장하면 전원에게 즉시 반영
- **권한 검사**: 모든 현장 요청에 로그인 ID 동봉 → 서버가 `accounts.json`으로 검사

## 1-3. AI 분석 아키텍처 — `ai-core.js` 단일 소스

AI 분석 로직(결정론 지표 + LLM 호출 + 프롬프트)은 **`ai-core.js` 한 파일**에만 있습니다. 서버 버전과 Pro 버전이 **동일한 이 파일**을 사용하므로, 분석을 고칠 때 한 곳만 수정하면 됩니다.

```
ai-core.js  ←── server.js (Enterprise: /api/ai/analyze 가 위임)
           ←── ai-server.js (Pro: node ai-server.js, 포트 3100)
```

| 항목 | 서버(Enterprise) | Pro(개인) |
|------|------------------|-----------|
| 실행 | `node server.js` (내장) | `node ai-server.js` (별도, 포트 3100) |
| 데이터 | 서버 `data/<siteId>.json` | 브라우저가 `inlineProj`로 요청에 동봉 |
| API 키 | 서버 `config.json` | 요청에 개인 키 동봉(브라우저 localStorage 보관) |
| 분석 엔진 | `ai-core.js` | `ai-core.js` (동일) |

**제공 지표**: 정식 CPM(Total Float·임계경로·예상준공/슬립), EVM(기성율·SPI·SV), 진척 추세(주당 속도·정체), 지연 무게·선후행 충돌·TF 급감·블라인드 스팟·병행 조합·준공 위험·신뢰도(confidence). 단일턴/멀티턴 모두 동일 지표 사용, 자기검증 2차 패스 적용.

**프롬프트(지시서)**: `prompts/Groq_Llama3.3-70B_작업지시서.md`(무료·빠름) / `prompts/Claude_작업지시서.md`(정밀). 최초 실행 시 없으면 자동 생성되며, provider에 따라 자동 선택됩니다. 환경변수 `AI_SELF_CRITIQUE=0`으로 자기검증 끄기 가능.

## 2. 권한(역할) 모델

| 역할 | 권한 |
|------|------|
| 마스터(master) | 전 현장 + 계정·현장 생성/수정/삭제 + 담당관리자 배정 + 백업/복원 |
| 담당관리자(manager) | 배정된 현장만 — 편집·직원 추가·공정회의·백업/복원 |
| 일반직원(staff) | 배정 현장 보기·편집·**공정회의** 가능 / 백업·복원은 **불가** |

- **편집 자동저장**은 전원 허용(공유 공정표 즉시 반영).
- **백업·복원(메뉴)** 은 마스터·담당관리자만. 직원은 💾 백업 버튼이 숨겨집니다.
- **관리자 탭 접근 코드**: 계정별 코드(마스터가 등록 시 공통 기본값 부여, 본인이 이름 클릭 → 변경). 하드코딩 코드는 최초 셋업(로그인 화면 5클릭) 전용.

## 3. 하도사 파일함 + 공정회의

### 3-1. 현장별 파일 폴더

```
하도업체/<현장명>/접수/   ← 메신저로 받은(또는 업로드한) 하도 파일
하도업체/<현장명>/보관/   ← 처리(배포) 완료 파일
```

- 현장 생성 시 자동 생성, **현장명 변경 시 폴더명도 자동 변경**.
- 전 구성원(마스터·담당관리자·직원)이 자기 현장 폴더에 업로드·조회·이동 가능(원격 포함).

### 3-2. 취합 탭 — 2단 파일함

- **드래그&드롭/클릭 업로드** → 원본이 **접수 폴더**로 업로드(+ 기존 매칭/저장 옵션 유지).
- 좌 📥 접수 / 우 🗄 보관, 체크박스 다중 선택, 가운데 **▶(보관으로)·◀(접수로)** 이동.
- **⬇ 선택 다운로드(최신본)** — 선택 회사의 현재(회의 반영) 공정표를 생성해 다운로드.

### 3-3. 취합 시 데이터 매핑

| 항목 | 원도급 유지 | 하도 반영 |
|------|------------|-----------|
| P 계획 시작·종료 | ✅ 원도급 기준 | — |
| A 실적 시작·종료 | — | ✅ 가져옴 |
| 실적율(%) | — | ✅ 가져옴 |
| 금액 | ✅ 원도급 유지 | ❌ 가져오지 않음 |
| 공종명·대분류 | 하도 파일의 공종명 그대로 | — |

> 원도급 금액과 하도 금액은 다릅니다. 취합은 진도 파악 목적이므로 금액은 가져오지 않습니다.

### 3-4. 공정회의(통합 비교) — 직원 포함 사용 가능

1. **업체 선택**: 접수/보관 파일의 회사명 목록 — **접수에 파일 있는 회사만 활성**(보관만 있으면 비활성). 접수가 비면 "접수된 자료가 없습니다".
2. 선택 회사들을 하나의 타임라인에 **적층 비교** → 공정 간섭 확인.
3. **🤖 AI 간섭분석** — 지연·충돌 + 만회 제안.
4. **📤 배포**: 참여사 최신 공정표를 `<회사명>_배포_날짜.json`으로 **보관 폴더에 저장 + 해당 접수 파일 제거**. 미참여사 접수 파일은 그대로 유지.

> ⚠️ `[📊 원도급사 통합 적용]` 은 원도급 현재 공정을 덮어쓰는 **구(舊)·위험 기능**입니다. 가급적 위 공정회의 흐름을 사용하세요.

## 4. 백업 / 복원 (서버 모드 — 원도급 주 공정표)

| 항목 | 내용 |
|------|------|
| 저장 위치 | `백업/<현장명>/` (현장 생성 시 자동 생성, 현장명 변경 시 폴더명도 변경) |
| 백업 대상 | 원도급 주 공정표 (하도는 접수/보관에서 관리하므로 제외) |
| 수동 백업 | 마스터·담당관리자 → **서버 백업폴더 + 내 PC 동시** 저장 |
| 자동 백업 | 복원 직전 / 매일 22:00(+07:00 누락 시) — **직전 백업과 내용 동일하면 파일 없이 로그만** |
| 파일명 | 수동 `<회사명>_백업_시각_작성자` / 자동 `_자동백업-복원전_`·`_자동백업-일일_` (자동 30개 유지) |
| 복원 | 백업 목록에서 선택 → **복원 직전 자동백업 후** 현재 공정표 교체 → 전원 자동 새로고침 |
| 이력 | `백업/<현장>/백업로그.jsonl` — 누가·언제·백업/복원/삭제·파일·종류 |
| 삭제 | **서버 PC(localhost)에서만** 가능 + 로그 기록 |

🛡️ **데이터 보호 가드**: 기존에 공종이 있는 현장을 **빈 데이터로 덮어쓰려는 저장은 서버가 차단**하고, 직전 파일을 `.protect_*`로 보호 스냅샷 + 로그를 남깁니다. (캐시 혼선 등으로 인한 의도치 않은 초기화 방지)

> 서버 복원은 **모든 접속자에게 일괄 적용**됩니다 — 근무 외 시간 권장.

## 5. 별도 창 (로컬 대조/작성)

- 서버 접속 중 취합 탭의 **🗔 로컬 파일(새 창)** / **🗔 새 공정표(새 창)** → 별도 윈도우에서 **로컬 전용** 편집(서버 미저장).
- 용도: 원도급 공정표(본창)와 배포본/대조본(새 창)을 **나란히 보며 실적·일자 수정**.
- 새 창은 로그인·서버 저장이 없고, 저장은 파일 다운로드. AI 분석은 **서버 키로** 동작(데이터만 인라인 전송).

## 6. 보안 모델 요약

| 저장 위치 | 보안 수준 |
|---|---|
| Pro(로컬) | 파일 기반 = PC 접근자 전체 공유(엑셀 수준) |
| 서버(Enterprise) | 계정 인증 + 사내망 — 가장 안전 |

### 민감 파일(절대 GitHub 업로드 금지 · `.gitignore` 처리)

- `config.json` — Claude/Groq API 키
- `accounts.json` — 사용자 계정·비밀번호·역할
- `data/`, `하도업체/`, `백업/` — 실제 공정·하도·백업 데이터
- 잔여물 `*.bak_*`, `*.corrupt_*`, `*.protect_*`, `*.empty_*`, `_스냅샷/`

### 건설현장 권장 구성

```
원도급사 서버(사내망)
  ├── 원도급 담당자만 서버 계정 보유
  ├── 직원도 배정 현장에서 편집·공정회의 가능(백업/복원은 불가)
  └── 하도 데이터는 접수 폴더 업로드(또는 .json 전달) → 취합·공정회의에서 처리

하도업체 각사
  └── Pro 버전 로컬 사용 → .json 파일을 원도급에 전달
```

## 7. 파일 구조

```
Terminus_master_schedule.html   ← 단일 파일 SPA (공개)
ai-core.js                 ← AI 분석 엔진(단일 소스) — server.js·ai-server.js 공용 (공개)
ai-server.js               ← Pro용 초경량 AI 서버 (node ai-server.js, 포트 3100) (공개)
server.js                  ← 서버 백엔드: 멀티현장·권한·파일함·백업/복원 (AI는 ai-core 위임) (공개)
prompts/Groq_Llama3.3-70B_작업지시서.md   ← Groq AI 지시서 (없으면 자동 생성, 공개)
prompts/Claude_작업지시서.md             ← Claude 정밀 분석 지시서 (없으면 자동 생성, 공개)
.gitignore                 ← 아래 비공개 파일 제외
─ 비공개(자동 생성) ─
accounts.json              ← 계정·역할·현장
data/<siteId>.json         ← 현장별 공정표
config.json                ← API 키
하도업체/<현장>/접수·보관/    ← 하도 파일
백업/<현장>/                ← 백업 + 백업로그.jsonl
```
